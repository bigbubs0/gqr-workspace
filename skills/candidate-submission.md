# Candidate Submission Email

---

## CRITICAL RULES

### Rule 1: No Narration

Output the email. Nothing else. No "here's your submission email," no "I've drafted the following," no summary of what you're about to do. Subject line first, body second, done.

### Rule 2: Sentence Editor Rules Apply to All Output

Every sentence in the email must pass all 8 rules before delivery:

1. **Impact endings** — most important word lands at the end of each sentence
2. **Define acronyms** on first use, abbreviate after (exception: biotech-native terms the client already knows — CDM, PV, GCP, CDISC, NDA, BLA, IND — never define these)
3. **Maximum economy** — fewest words that lose nothing
4. **No repetition** — never restate information already in a previous sentence
5. **Vary word choice** — no word appears twice in the same paragraph unless unavoidable
6. **One advanced word per sentence max**
7. **Two-comma max** — if a sentence needs 3+ commas, split it
8. **Kill adverbs** — replace with a stronger verb or adjective

### Rule 3: One Question Max

If critical information is missing, ask ONE question that covers every gap. Never ask sequentially. Never ask about optional fields (comp, visa) — skip those sections if unknown rather than asking.

### Rule 4: Speed Is the Priority

This is Bryan's highest-volume task. On team/split accounts, 8 teammates compete the moment a role drops. Do not ask for information that isn't essential to send the email. If a reasonable inference exists, make it and deliver. Flag the assumption at the bottom of the output in a single bracketed note — only if it materially affects the email.

---

## Input

Bryan provides some combination of:

- **Candidate profile** — CV, Notion candidate record, LinkedIn, verbal summary, or screening notes
- **Target role** — role title, client name, or link to an Active Search in Notion
- **Context** — comp expectations, availability, relocation, visa status, contract vs. perm preference, competing offers, or client-specific context

Minimum viable input to produce the email: candidate name + current title/company + role title + client name. If even these are missing, ask one question.

---

## Context Loading

Before drafting, load the following context silently. Do not reference the files unless asked.

1. **program.md** — active account state, fee structure (Owned 100% vs. Team 50%), client contacts, current pipeline
2. **Notion Active Searches** — verify the role is still active (not On Hold, Closed, or Filled). If status is unclear, note it at the bottom of the output rather than stopping the draft.
3. **Notion Candidates** — check for an existing candidate record. Pull key skills, accomplishments, comp data, and availability if present.

**Fee structure check:** Owned (100%) = full priority. Team (50%) = speed-first; submit immediately and surface competing candidates after. The email itself does not change by fee type — only the priority urgency changes.

## Email Structure

Produce output in this exact order:

```
Subject: Candidate Submission - [Candidate Name] - [Role Title]

Hi [Client Contact Name],

[Opening line]

[Candidate summary block]

[Comp block — omit entirely if unknown]

[Availability block]

[Work authorization block — omit if US citizen or not relevant]

[Closing line]

Bryan Blair | Vice President, Clinical | GQR Global Markets
```

---

## Section-by-Section Rules

### Subject Line

Format exactly as:

```
Candidate Submission - [Candidate Name] - [Role Title]
```

Use the role title as it appears in the Active Search or as Bryan states it. Do not abbreviate or editorialize.

---

### Opening Line

One sentence. Sets context without restating the subject line. Options depending on what Bryan provides:

- If Bryan has a relationship with this contact: "Following our conversation last week, I wanted to get [First Name]'s background in front of you."
- If this is a fresh submission: "I have a strong candidate for your [Role Title] search."
- If Bryan gave specific context (e.g., client mentioned urgency, specific need): Use that context. "You mentioned needing oncology protocol experience — [Candidate First Name] has built that background across the last [X] years."

One sentence. No "I'm excited to share" or "I'm pleased to present." No generic opener.

---

### Candidate Summary Block

**Header line:**

```
[Full Name] | [Current Title], [Current Company] | [X] years of experience
```

If the candidate is a contractor or consultant, note it: "Contract [Title]" or "Independent Consultant."

**Bullets:** 3-5 bullets. Each bullet must:

- Map one candidate qualification to one specific role requirement
- Contain one concrete data point: company name, years, trial phase, headcount, system name, therapeutic area, submission type, or outcome
- Begin with the qualification, end on the impact or differentiator

**Bullet construction — required format:**

```
- [Qualification]: [Specific evidence — company, scope, outcome]
```

**Prohibited in bullets:**
- "Extensive experience" — replace with years or scope
- "Proven track record" — replace with a specific result
- "Results-oriented" — replace with a result
- "Strong background in" — replace with the background itself
- "Expertise in" — replace with the expertise itself
- Any bullet that could apply to a different candidate in a different search

**Mapping rule:** Read the role requirements (from Active Search notes or JD if Bryan provides one). Every bullet should answer a specific requirement. If Bryan provides no JD, use the role title and Bryan's verbal context to infer the top requirements. Biometrics roles care about CDISC/software depth. PV roles care about case volume, submission types, regulatory experience. Clinical Development VP roles care about phase, therapeutic area, FDA submission, and team scope. CDM roles care about EDC systems, CDISC compliance, and trial scale.

---

### Compensation Block

**Include only if Bryan provides comp data.** Do not write "Compensation: TBD" or "Compensation: Not provided." Skip the section entirely if unknown.

When included:

```
Compensation: [Current base or rate] current. [Target or expectation] target.
```

For contractors: use hourly rates. Include all-in bill rate if known.

```
Compensation: $[X]/hr pay rate. $[Y]/hr all-in bill rate.
```

---

### Availability Block

Always include. Format:

```
Availability: [Start date or notice period]. [Contract/perm preference if relevant.]
```

Examples:
- "Availability: Available immediately."
- "Availability: Two-week notice period. Available [estimated date]."
- "Availability: Open to contract or perm."
- "Availability: Contract preferred. Available [date]."

If Bryan doesn't provide availability: write "Availability: Flexible — happy to confirm timeline on a call."

---

### Work Authorization Block

Include only when relevant — H-1B, OPT, TN, or other visa situations the client needs to know. If the candidate is a US citizen or green card holder and no visa complexity exists, skip this section entirely.

When included, one line:

```
Authorization: [Status — H-1B transfer required / TN visa / OPT expires [date] / etc.]
```

Do not editorialize on visa risk in the client email. Flag concerns in a post-email note to Bryan if relevant.

---

### Closing Line

One sentence suggesting a next step. Use one of these patterns — vary across submissions:

- "Happy to set up a call to discuss [First Name]'s background in more detail."
- "Let me know if you'd like to move forward with an interview."
- "Available to connect whenever works for your team."
- "I can have [First Name] available for a call as early as [timeframe]."

No "please don't hesitate to reach out." No "feel free to." No "I look forward to hearing from you."

---

### Signature

Always exactly:

```
Bryan Blair | Vice President, Clinical | GQR Global Markets
```

---

## Quality Gates

Before delivering, verify every item:

- [ ] Subject line matches format exactly
- [ ] Greeting uses client's actual first name (not "[Client Name]")
- [ ] Every bullet maps to a specific role requirement
- [ ] Every bullet contains at least one concrete data point
- [ ] No prohibited phrases ("extensive experience," "proven track record," "results-oriented," "strong background in")
- [ ] Comp section is either populated with real numbers or omitted entirely
- [ ] Availability is present in every email
- [ ] Work auth section is present only when relevant
- [ ] Email body is 150-250 words (subject line and signature excluded)
- [ ] No sentence has more than 2 commas
- [ ] No adverbs
- [ ] No em dashes — hyphens only
- [ ] Signature block is exact
- [ ] Zero placeholder brackets remain in the output

---

## Length Standard

**Email body: 150-250 words.** Count from the opening line through the closing line. The header line and bullets are the core — keep prose minimal. Clients read fast. The CV is attached; the email sells the meeting.

## Common Patterns by Role Type

Use these as the qualification-to-requirement mapping lens when Bryan provides minimal role context:

| Role | Top 3 requirements to map against |
|------|-----------------------------------|
| VP/SVP Clinical Development | Phase ownership (II/III/IV), therapeutic area, NDA/BLA submission, team scale |
| VP/SVP Drug Safety/PV | Case volume, submission types (aggregate/expedited), regulatory inspections, vendor oversight |
| Director/VP Biostatistics | CDISC SDTM/ADaM expertise, SAS/R proficiency, NDA/BLA table/listing/figure accountability, team size |
| Director/VP CDM | EDC platform depth, CDISC/CDASH compliance, trial scale (# of studies), CRO oversight |
| Sr. CTM / CTM (Perm or Contract) | Phase II/III protocol execution, CRO oversight, site management, budget accountability |
| CRA (Sr. or Staff) | Monitoring experience (sponsor-side), therapeutic area, GCP compliance, site count managed |
| Medical Director / Sr. MD | MD degree, therapeutic area, protocol authorship, safety review experience |
| CMO / C-Suite R&D | Board/leadership visibility, NDA/BLA portfolio, therapeutic area breadth, team scale |

## What This Skill Is NOT

- Not a CV rewriter. The CV is the attachment. The email sells the call.
- Not a CLABVISSS tool. Candidate control and closing strategy are separate from the submission email.
- Not a place to editorialize. The email presents facts mapped to requirements. Opinions about the candidate's fit belong in verbal conversation with the client.
- Not an interview prep tool. Candidate prep is a separate skill.
